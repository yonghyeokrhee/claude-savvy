# Agent Team — 에이전트 팀

## 에이전트 팀이란?

**Agent Team**은 여러 개의 Claude 인스턴스가 하나의 팀으로 협력하여 복잡한 작업을 수행하는 아키텍처입니다.

기존 Sub-Agent가 "상사가 부하에게 일을 시키고 보고받는" 관계였다면, Agent Team은 **동료들이 서로 대화하고 조율하며 함께 일하는** 관계입니다.

## Sub-Agent와 Agent Team의 차이

이전 강의에서 배운 Sub-Agent는 메인 Claude에 종속된 관계였습니다. Agent Team은 한 단계 더 나아갑니다:

| 구분 | Sub-Agent | Agent Team |
|---|---|---|
| **컨텍스트** | 독립 컨텍스트, 결과만 반환 | 독립 컨텍스트, 완전히 자율적 |
| **커뮤니케이션** | 메인 Claude에만 보고 | 팀원끼리 직접 대화 가능 |
| **조율 방식** | 메인 Claude가 모든 작업 관리 | 공유 태스크 리스트로 자기 조율 |
| **적합한 상황** | 결과만 필요한 집중 작업 | 토론·협업이 필요한 복잡 작업 |
| **토큰 비용** | 낮음 (요약 반환) | 높음 (각 팀원이 별도 인스턴스) |

```
Sub-Agent 모델:
┌──────────────┐
│  메인 Claude  │
└──┬───┬───┬───┘
   ↓   ↓   ↓
  S1  S2  S3     ← 결과만 위로 반환 (단방향)

Agent Team 모델:
┌──────────────┐
│   Team Lead   │
└──┬───┬───┬───┘
   ↕   ↕   ↕
  T1 ↔ T2 ↔ T3   ← 서로 대화 가능 (양방향)
   └───┴───┘
   공유 태스크 리스트
```

## 에이전트 오케스트레이션 패턴

2026년 현재, AI 에이전트 시스템에는 다섯 가지 주요 아키텍처 패턴이 있습니다:

### 1. Orchestrator-Worker (오케스트레이터-워커)

가장 널리 쓰이는 패턴입니다. 중앙의 오케스트레이터가 작업을 분해하고, 워커에 할당하고, 결과를 취합합니다.

```
사용자 요청
    ↓
┌───────────────┐
│  Orchestrator  │  ← 작업 분해 + 결과 취합
└──┬────┬────┬──┘
   ↓    ↓    ↓
  W1   W2   W3      ← 각 워커가 독립 실행
```

Claude Code의 Sub-Agent가 이 패턴입니다. Anthropic의 리서치 시스템도 이 패턴을 사용하여, Lead Agent가 3~5개 Sub-Agent를 병렬 실행하고 결과를 종합합니다.

### 2. Swarm (스웜)

중앙 지휘자 없이 에이전트들이 **로컬 규칙**에 따라 자율적으로 협력합니다. 각 에이전트가 독립적으로 행동하지만, 전체적으로는 하나의 목표를 향해 움직입니다.

```
    A1 ←→ A2
    ↕  ╲╱  ↕
    A3 ╱╲ A4    ← 공유 상태를 통해 간접 조율
    ↕  ╱╲  ↕
    A5 ←→ A6
```

OpenAI의 Agents SDK가 Swarm 패턴에서 출발했습니다. 에이전트 간 "핸드오프"를 통해 작업을 넘기는 방식입니다.

### 3. Hierarchical (위계적)

트리 구조로 에이전트를 조직합니다. 상위 매니저가 중간 감독자에게, 감독자가 실무 에이전트에게 위임하는 방식입니다.

```
        CTO Agent
       ╱         ╲
  Frontend Lead  Backend Lead
   ╱    ╲        ╱    ╲
 UI    Test    API   DB Agent
Agent  Agent  Agent
```

대규모 소프트웨어 프로젝트처럼 역할이 명확히 나뉘는 작업에 적합합니다.

### 4. Pipeline (파이프라인)

작업이 순차적으로 한 에이전트에서 다음 에이전트로 흐르는 패턴입니다. 각 단계가 이전 결과를 입력으로 받습니다.

```
요구사항 분석 → 설계 → 코딩 → 리뷰 → 테스트
   Agent 1    Agent 2  Agent 3  Agent 4  Agent 5
```

CI/CD 파이프라인과 유사하며, 단계별 품질 관리가 중요한 작업에 적합합니다.

### 5. Mesh (메시)

모든 에이전트가 동등하게 직접 통신합니다. Swarm과 비슷하지만, 공유 상태 대신 **직접적인 P2P 메시징**을 사용합니다.

```
A1 ←→ A2 ←→ A3
 ↕  ╲    ╱  ↕
 ↕   ╲  ╱   ↕
A4 ←→ A5 ←→ A6
```

## Claude Code Agent Team 실전

### 활성화

Agent Team은 실험적 기능으로 기본 비활성화 상태입니다. `settings.json`에서 활성화합니다:

```json
{
  "env": {
    "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"
  }
}
```

### 팀 생성하기

자연어로 팀 구성을 요청하면 됩니다:

```
CLI 도구를 설계하려고 해. 에이전트 팀을 만들어줘:
- UX 담당 팀원 1명
- 기술 아키텍처 담당 1명
- 반론을 제기하는 Devil's Advocate 1명
```

Claude가 팀을 생성하고, 팀원을 배정하고, 공유 태스크 리스트를 만들어 작업을 조율합니다.

### 팀 구조

| 구성 요소 | 역할 |
|---|---|
| **Team Lead** | 팀을 생성하고 작업을 배정하는 메인 세션 |
| **Teammates** | 각자 독립된 Claude Code 인스턴스 |
| **Task List** | 팀원들이 공유하는 작업 목록 (대기/진행중/완료) |
| **Mailbox** | 에이전트 간 메시지 시스템 |

### 디스플레이 모드

팀원들의 작업을 확인하는 두 가지 방식이 있습니다:

- **In-process**: 하나의 터미널에서 `Shift+Down`으로 팀원 순회. 별도 설치 불필요
- **Split panes**: tmux 또는 iTerm2로 각 팀원을 별도 패널에 표시. 동시에 모든 팀원 출력 확인 가능

### 실전 시나리오

**PR 병렬 리뷰:**

```
PR #142를 에이전트 팀으로 리뷰해줘:
- 보안 취약점 전문가 1명
- 성능 영향 분석가 1명
- 테스트 커버리지 검증자 1명
각자 리뷰 후 결과를 종합해줘.
```

**경쟁 가설 디버깅:**

```
앱이 메시지 하나 보내면 연결이 끊기는 문제가 있어.
5명의 팀원을 만들어서 각자 다른 가설을 조사해줘.
서로 토론하면서 상대 이론을 반증해보고,
합의된 결론을 문서로 정리해줘.
```

이 "토론 구조"가 핵심입니다. 순차 조사는 처음 발견한 원인에 편향되지만, 독립적인 조사원들이 서로 반증하면 진짜 원인을 찾을 확률이 높아집니다.

## Anthropic 리서치 시스템 사례

Anthropic은 자사 리서치 시스템에 멀티 에이전트 아키텍처를 적용했습니다:

| 항목 | 내용 |
|---|---|
| **아키텍처** | Orchestrator-Worker (Lead Agent + Sub-Agents) |
| **Lead Agent** | Claude Opus — 쿼리 분석, 전략 수립, 결과 종합 |
| **Sub-Agents** | Claude Sonnet — 각각 독립된 리서치 수행 |
| **병렬화** | 에이전트 레벨(3~5개 병렬) + 도구 레벨(3개+ 병렬) |
| **성능** | 단일 에이전트 대비 **90.2% 성능 향상** |
| **리서치 시간** | 복잡한 쿼리 기준 최대 **90% 단축** |
| **토큰 사용** | 채팅 대비 약 15배 |

### 리소스 할당 가이드

Anthropic이 제시하는 에이전트 규모 기준:

| 작업 유형 | 에이전트 수 | 도구 호출 |
|---|---|---|
| 단순 팩트 검색 | 1개 | 3~10회 |
| 직접 비교 분석 | 2~4개 | 각 10~15회 |
| 복잡한 리서치 | 10개+ | 역할별 명확 분배 |

## 2026년 멀티 에이전트 트렌드

2025년이 단일 AI 어시스턴트의 해였다면, 2026년은 **에이전트 팀의 해**입니다:

- Gartner 예측: 2026년 말까지 기업 앱의 **40%가 멀티 에이전트 시스템** 채택
- Anthropic Agentic Coding Report: "멀티 에이전트 시스템이 단일 에이전트 워크플로우를 대체"
- 주요 프레임워크: Claude Agent Teams, OpenAI Agents SDK, LangGraph, CrewAI, Swarms, AutoGen

핵심 원칙은 명확합니다: **중앙 집중형으로 시작하고, 구체적인 병목이 생길 때만 분산하라**. 대부분의 프로덕션 시스템은 Orchestrator-Worker 패턴에서 출발합니다.

## 주의사항

- Agent Team은 **실험적 기능**입니다 — 세션 재개, 태스크 조율, 종료 동작에 제한이 있습니다
- 토큰 비용이 **팀원 수에 비례**하여 증가합니다 — 3~5명으로 시작하세요
- 팀원 당 **5~6개 태스크**가 적절한 분배입니다
- 같은 파일을 여러 팀원이 수정하면 **충돌**이 발생합니다 — 파일 소유권을 명확히 나누세요
- Agent Team에는 현재 **Opus 4.6 모델이 필수**입니다
